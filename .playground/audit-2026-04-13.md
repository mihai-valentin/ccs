# CCS Project Audit тАФ 2026-04-13

## Summary

The codebase is well-structured for its age. The core data pipeline (scan -> parse -> upsert -> query) is solid and the indexer/DB layers have reasonable test coverage. The major recurring themes are: **systemic code duplication across packages**, **a shell injection risk in the opener**, **TUI delete not removing JSONL files**, **silent migration error suppression**, and **N+1 query patterns in the TUI**.

---

## Critical (3)

### C-1 тАФ Shell injection via `session.Cwd` and `session.ID` in OpenSession

**Category:** code/security
**File:** `internal/opener/opener.go:24`

```go
shellCmd := fmt.Sprintf("cd %q && claude --resume %q", session.Cwd, session.ID)
cmd := exec.Command("bash", "-c", shellCmd)
```

`%q` applies Go double-quote escaping, not POSIX shell escaping. A `Cwd` containing `$()`, backticks, or `\` can escape the quoted string and execute arbitrary commands. Since both values come from files on disk, this is a real injection vector.

**Fix:** Use `exec.Command` directly without `bash -c`:

```go
cmd := exec.Command("claude", "--resume", session.ID)
cmd.Dir = session.Cwd
```

---

### C-2 тАФ TUI delete does not remove the JSONL file from disk

**Category:** code
**File:** `internal/tui/app.go:251`

`updateDeleteConfirm` only calls `m.db.DeleteSession(session.ID)`, which removes the DB row. The CLI `delete` command additionally calls `os.Remove(jsonlPath)` (`internal/cli/delete.go:46-49`). A TUI-deleted session re-appears on the next invocation because the JSONL file still exists and gets re-indexed.

**Fix:** After `DeleteSession` succeeds, construct the JSONL path and call `os.Remove`. Thread `claudeDir` through `NewModel` or store it on the `Model` struct.

---

### C-3 тАФ Silent schema migration error swallowed

**Category:** code
**File:** `internal/db/db.go:54`

```go
sqlDB.Exec("ALTER TABLE sessions ADD COLUMN summary TEXT")
```

The error return is discarded. If this fails for any reason other than "column already exists" (locked DB, corrupted schema), the tool continues with a broken DB state.

**Fix:** Check the error and only ignore "duplicate column name":

```go
_, err = sqlDB.Exec("ALTER TABLE sessions ADD COLUMN summary TEXT")
if err != nil && !strings.Contains(err.Error(), "duplicate column name") {
    sqlDB.Close()
    return nil, fmt.Errorf("migrate summary column: %w", err)
}
```

## High (6)

### H-1 тАФ Unbounded N+1 query loop at TUI startup

**Category:** code/performance
**File:** `internal/tui/app.go:75-78`

`NewModel` fires one `GetSessionTags` query per session inside a loop. With 1000 sessions (the TUI's hard limit), that's 1001 sequential SQLite round-trips before the screen renders.

**Fix:** Add a `GetAllSessionTags() (map[string][]model.Tag, error)` bulk query.

### H-2 тАФ `refreshTags` in TUI also fires an N+1 loop on every tag operation

**Category:** code/performance
**File:** `internal/tui/app.go:441-450`

Same issue as H-1 тАФ `refreshTags` re-queries every session's tags after each add/remove.

### H-3 тАФ Systemic code duplication across packages

**Category:** arch
**Files:** Multiple

| Function | Duplicated In |
|---|---|
| `formatRelativeTime` | `cli/helpers.go:61`, `tui/list.go:111` |
| `sessionDisplayName` | `cli/helpers.go:104` (2-arg), `tui/app.go:462` (1-arg) |
| `truncate` / `truncateStr` | `cli/helpers.go:97`, `tui/list.go:132`, `indexer/parser.go:183` |
| `parseTime` | `db/queries.go:18`, `opener/resolve.go:200` **(diverged!)** |
| `extractContent` + JSONL structs | `indexer/parser.go:146`, `summarizer/summarizer.go:160` |

The `parseTime` divergence is dangerous: `opener/resolve.go` only tries `time.RFC3339` (no nanosecond fallback), so sessions stored with sub-second precision get their timestamps silently zeroed when resolved by name/prefix.

**Fix:** Consolidate into `internal/format` or `internal/timeutil`. Move `extractContent` + JSONL structs to `internal/indexer` and import from `summarizer`.

### H-5 тАФ TUI hard-codes a 1000 session cap with no indication

**Category:** ux
**File:** `internal/cli/ui.go:26`

Users with >1000 sessions get a silently truncated list. No "showing N of M" indicator. `summarize --all` uses `Limit: 10000`, making the inconsistency visible.

### H-6 тАФ `detectProjectDir` path sanitisation doesn't match Claude Code's logic on all platforms

**Category:** code
**File:** `internal/cli/list.go:104-110`

**Fix:** Always replace `/` with `-`, not `os.PathSeparator`.

## Medium (9)

### M-1 тАФ `ListSessions` tag JOIN uses OR semantics, not AND

**Category:** code
**File:** `internal/db/queries.go:128-137`

With multiple `--tag` flags, the query returns sessions with **any** of the tags, not **all** of them. Users expect AND semantics from repeatable flags.

**Fix:** Use `HAVING COUNT(DISTINCT t.name) = N` grouped subquery.

### M-2 тАФ `PurgeMissingSessions` builds an unbounded SQL `IN` clause

**Category:** code
**File:** `internal/db/queries.go:309-326`

With thousands of sessions, the `NOT IN (?, ?, ... x5000)` may exceed SQLite's `SQLITE_LIMIT_VARIABLE_NUMBER`.

**Fix:** Use a temp table or batch the `NOT IN` list in chunks.

### L-10 тАФ Upsert overwrites summary to NULL on reindex

**Category:** code
**File:** `internal/db/queries.go:42-56`

The `ON CONFLICT DO UPDATE` sets all fields including those not in the upsert args. While summary isn't in the INSERT column list (so it stays NULL for new rows), the issue is that `upsertSessionSQL` updates every listed column. Currently summary is NOT in the upsert list so it's preserved тАФ but this is fragile. If anyone adds summary to the upsert column list in the future, all summaries get wiped on reindex.

**Recommendation:** Add a comment documenting this intentional omission, or change the upsert to explicitly use `summary = COALESCE(sessions.summary, excluded.summary)`.
